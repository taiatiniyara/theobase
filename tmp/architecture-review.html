Theobase was built fast — 30+ commits across T1–T14 and the #-series, all features landed, all tests green. But that velocity came at a cost: the codebase accumulated patterns that make the next feature harder than the last. Below are five deepening opportunities, ranked by impact on testability and agent-navigability.

Candidate 1

Extract a Data Access Layer

worker/routes/*.ts (all 12 files), worker/index.ts

Strong

Problem

Every route handler mixes HTTP concerns (request parsing, response building, permission checks) with raw D1 SQL. members.ts is 1,803 lines with 22 handlers each writing their own queries. There is zero separation between "what the caller asked for" and "how it's stored."

  • ~130 env.DB.prepare() calls scattered across 12 files
  • Repeated existence checks ("does this church exist?") in dozens of handlers
  • Dynamic query building (string concatenation of WHERE clauses) in 6 handlers
  • Business logic (e.g., tithe forwarding rules) interleaved with INSERT statements
improves locality enables unit tests new seam

Solution

Introduce per-entity repository modules behind a consistent interface. Each repository encapsulates all SQL for one domain entity (Member, Church, Transaction, etc.) and returns typed objects — not raw D1 results. Route handlers call repository methods and focus purely on HTTP concerns. 

// Before: inline SQL in route handler
const result = await env.DB.prepare(
  "SELECT m.*, h.name as household_name FROM members m LEFT JOIN households h ON m.household_id = h.id WHERE m.church_id = ? AND m.status = ?"
).bind(churchId, status).all();

// After: repository call
const members = await memberRepo.findByChurch(churchId, { status });

Benefits

Locality

All member queries live in one repository. Change the query shape once. The deletion test: removing MemberRepo spreads 22 queries across 12 files.

Leverage

Every route handler gets simpler — one repository method call instead of raw SQL + bind + all + map. Conference-wide queries benefit from shared indexing.

Testability

Repositories are testable with an in-memory SQLite adapter. Route handlers test with a fake repository — no D1 needed in unit tests.

Candidate 2

Request Pipeline Router

worker/index.ts (536 lines, ~100 if-blocks)

Worth Exploring

Problem

The router is a flat imperative if/else chain — no prefix grouping, no middleware pipeline, no parameter extraction standard. Rate-limiting is the only "middleware" and it's called manually per-route. Every new route adds more lines to one function.

  • 536-line fetch() handler with ~100 if-blocks
  • Route parameters extracted via regex capture groups, cast to Number()
  • No way to apply middleware to a group (e.g., "all /api/members/* needs members:read")
  • CORS handled once at top, everything else is manual
improves leverage new seam

Solution

A lightweight composable router (no framework dependency — ~80 lines of code) that supports prefix grouping, method dispatch, parameter extraction, and middleware chains. Built around a Router class with .use() for middleware and .get()/.post()/.patch() for handlers. 

// Before: flat if/else
if (path === "/api/v1/finance/batches" && request.method === "GET") {
  const auth = await authenticate(request, env);
  if (auth instanceof Response) return auth;
  const forbidden = authorize(auth, PERMISSIONS["finance:read"]!);
  if (forbidden) return forbidden;
  return handleGetBatches(request, env);
}

// After: composable router
router.group("/api/v1/finance", [authenticate, authorize("finance:read")], (r) => {
  r.get("/batches", handleGetBatches);
  r.get("/batches/:id", handleGetBatch);
  r.post("/batches", handleCreateBatch);
  r.post("/batches/:id/confirm", handleConfirmBatch);
});

Benefits

Locality

Middleware for a group lives at the group level — no repeating auth gates 55 times. Add a new middleware once, apply to all routes in a group.

Leverage

Route parameters are automatically extracted and passed to handlers. No more regex capture groups and Number() casts in the router body.

Testability

Router is an isolated module — test route matching and middleware execution without starting the worker.

Candidate 3

Frontend Shared Component Library

src/routes/*.tsx (22 page files, ~5,800 lines)

Worth Exploring

Problem

There is no src/components/ directory. Every page builds its UI from raw Tailwind classes repeated inline. The same tab row, form field, status badge, error alert, and table pattern is re-implemented in every file that needs it.

  • SummaryCard duplicated identically across 3 dashboard pages (14 lines each)
  • Tab-switching UI duplicated in 5 pages (identical CSS, just different tab labels)
  • Error/message alert divs duplicated ~35 times
  • Same form input styling block repeated in every single form field
  • No shared table component — same header/body/empty-state pattern in 6 files
improves locality improves leverage

Solution

Extract shared UI primitives into src/components/ — small, composable modules with a minimal interface. Each component handles one responsibility: display states, styling, and accessibility. Pages compose them. 

// Components to extract:

Candidate 4

Unify the json() Response Helper

All 13 files that define json() identically

Strong

Problem

The json() helper — a 5-line function that wraps Response with JSON content-type and CORS headers — is defined identically in 12 route files plus index.ts. This is the clearest DRY violation in the codebase. Changing CORS behavior or adding a header to all responses requires touching 13 files.

improves locality

Solution

Move json() into worker/lib/ alongside the other shared utilities. All 13 call sites import it instead of defining their own copy. 

// Move to worker/lib/response.ts
export function json(data: unknown, status = 200): Response {
  return new Response(JSON.stringify(data), {
    status,
    headers: { "Content-Type": "application/json", "Access-Control-Allow-Origin": "*" },
  });
}

// Routes import it:
import { json } from "../lib/response";
Candidate 5

Complete Durable Object Implementations

worker/durables/ConferenceDO.ts, worker/durables/ChurchSyncDO.ts

Speculative

Problem

ConferenceDO.getInfo() returns hardcoded "default" values — it doesn't actually aggregate data from child churches. ChurchSyncDO stores operation payloads as opaque strings without understanding their semantics. Both DOs exist and are registered but are effectively stubs.

  • getInfo() returns { name: "default", code: "default", churchCount: 0, memberCount: 0 } regardless of actual data
  • applyOfflineOperation() deduplicates by UUID but never validates the payload against the schema
  • No integration between ConferenceDO aggregates and the route-level reporting queries
completes seam

Solution

Wire ConferenceDO.getInfo() to query the D1 database for real church counts and member counts in the conference. Enhance applyOfflineOperation() to validate operation types against a known set and parse payloads before storing. This is speculative because the current architecture already handles these concerns through routes — the DOs would optimize rather than enable.

Top Recommendation

Candidate 1: Extract a Data Access Layer

This is the highest-leverage deepening opportunity. The current architecture has zero separation between HTTP handling and data access — every one of 72 route handlers mixes parsing, validation, authorization, SQL, and response formatting into a single function. Extracting repositories behind a clean interface would:

Reduce route complexity

Handlers drop from ~150 lines to ~40 lines. Focus on "what was asked" and "what to return" — not "which table and which columns."

Enable targeted testing

Repositories testable with an in-memory SQLite adapter. Route handlers testable with a fake. Currently: only integration tests against full miniflare runtime.

Unblock Candidates 2–5

A router makes more sense when handlers are 40 lines, not 400. Components make more sense when pages compose repositories, not raw SQL.

Start with one entity (Member, the largest at 1,803 lines) as a proof of concept. The repository interface becomes the seam — and as the codebase design principle says: "the interface is the test surface."
